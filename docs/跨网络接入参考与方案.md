# MACFlow 跨网络接入参考与可借鉴方案

## 1. 目标

让管理员在任意网络环境下，稳定进入 iStoreOS（面板/LuCI/SSH），同时尽量不暴露公网高风险端口。

## 3. 可借鉴要点

- Tailscale 子网路由可把不能装客户端的设备纳入统一访问域，适合“路由器作为入口”。
- Tailscale 默认依赖 NAT 穿透 + DERP 回退，跨复杂网络稳定性较高。
- Tailscale 对出口与中继端口有明确建议（TCP 443、UDP 41641 出站、UDP 3478 出站等）。
- Cloudflare Tunnel 的核心价值是“仅出站连接”，不需要给路由器开公网入站端口。
- OpenWrt 已有 `tailscale` 与 `ddns-scripts` 生态，便于在 iStoreOS 直接落地。

## 4. 建议的落地优先级

### A. 首选：Tailscale（推荐）

适用：你要“换网络也能进路由器”，并且希望运维复杂度低。

建议：

- 路由器加入 tailnet，固定一个可记忆域名（MagicDNS/机器名）。
- 管理端统一用该域名访问 MACFlow 与 SSH。
- 禁止直接暴露 MACFlow 的公网端口（18080）。

OpenWrt 常用起步（按包可用性调整）：

```sh
opkg update
opkg install tailscale
/etc/init.d/tailscale enable
/etc/init.d/tailscale start
tailscale up
```

### B. 次选：DDNS + VPN（WireGuard）

适用：已有固定域名策略、可接受自维护 VPN。

建议：

- 用 `ddns-scripts` 维护域名解析。
- 管理流量先走 VPN，再访问 LuCI/MACFlow。
- 不建议把 18080 直接暴露公网。

### C. 备选：Cloudflare Tunnel（Web 管理）

适用：只需远程 Web 管理，且希望“零公网入站”。

建议：

- 将 MACFlow/LuCI 通过 Tunnel 暴露在受控域名。
- 配 Access/身份认证策略后再开放给管理员。
- SSH 建议仍走 Tailscale/WireGuard，不建议直接穿隧暴露。

## 5. 与当前仓库联动建议

已落地的 `vm/quick-connect.sh` 支持：

- 本地+网关自动发现。
- 显式入口优先（`ISTORE_PANEL_URLS` / `ISTORE_LUCI_URLS` / `ISTORE_SSH_TARGETS`）。
- 配置文件加载（`~/.config/macflow/connect.env`）。
- WSL SSH 回退（Windows 下常见 2222 不可达场景）。

因此你可以把 Tailscale/DDNS/Cloudflare 的固定入口写进 `connect.env`，脚本会优先命中这些“跨网络稳定入口”。

## 6. 安全基线（建议默认）

- 不直接在公网开放 `:18080`。
- 远程入口至少加一层身份控制（Tailnet ACL、Access Policy、VPN）。
- SSH 禁止弱口令，建议仅密钥或最少可用面。
- 保留本地应急入口（LAN 网关地址 + 串口/本地 SSH）。
